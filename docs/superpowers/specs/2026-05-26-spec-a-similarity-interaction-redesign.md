# Spec A：相似度分析的交互重设计

- 日期：2026-05-26
- 作者：与用户结对设计
- 状态：待实现
- 关联：Spec B（扫描期预计算文件特征）

## 背景与动机

「相似度分析」是责任认领体验里关键的一环：用户电脑上同件事经常有多个文件（不同版本的合同、不同后缀的同一份方案），逐个认领体验差。当前实现通过 `BuildFamilies` 把相似文件聚成 family，列表里折叠展示，让用户只需对 primary 行操作。

但当前交互有三个具体问题：

1. **字面与价值脱节**：右上角「相似度分析」按钮像运维操作，用户看不出它跟"少点几次认领"的关系。
2. **时机错位**：用户必须先手动点按钮跑完分析，才能享受家族折叠的好处；属于"功能找用户"。
3. **隐性 bug**：当前列表 `GroupByFamily=true` 折叠掉非 primary，但 `BatchClaim` 不级联 family —— 等于非 primary 永远没机会被认领。

本 spec 通过两件事解决：
- 把"分析"从用户感知里抹去，做成"扫描完后台自动跑"（依赖 Spec B 让自动跑足够快）
- 把"批量认领时如何处理家族"显式化为弹窗交互，由用户授权扩展认领范围

## 范围与边界

**做：**
- 扫描完成时自动触发 BuildFamilies
- 认领列表的 primary 行新增"关联 N ▾" chip
- 单文件认领 / 批量认领两种弹窗形态
- 顶部「相似度分析」按钮改名「重建相似关系」+ 二次确认对话框
- 新增 `POST /analyze/preview` 接口给二次确认对话框算 N 和 M
- 新增 system_config 两项（默认 policy + 跳过弹窗开关）
- 系统设置页新增「相似认领默认行为」卡片

**不做：**
- 不动 BuildFamilies 算法（Spec B 只改数据来源）
- 不动 BatchClaim API 契约（前端把 IDs 列表算好直接传）
- 不动 `data_resources.is_claimed/claim_status/claimant_*` 字段
- 不动现有 `GroupByFamily` 折叠逻辑
- 不实现"自动级联认领"（必须经用户授权）
- 不实现增量 BuildFamilies（仍为全量重建，Spec B 让全量重建变快）

## 设计

### 1. 触发链路

```
扫描完成（首次普查 / 增量扫描 / 新数据登记）
   ↓ 后端钩子自动调用
runAnalysis（现有 internal/httpd/similarity.go）
   ↓ 完成后写 data_resources.family_*
用户进入认领页
   ↓
看到家族 chip，零等待
```

实现：在扫描完成的后端钩子里追加 `go runAnalysis(...)` 异步调用。复用现有 `runMu` 互斥锁防并发；如果已有任务在跑，新增任务直接跳过（无需排队）。

### 2. 认领页主界面变更

| 元素 | 现在 | 改后 |
|---|---|---|
| 列表（`GroupByFamily=true` 折叠） | 显示 primary 行 + 三个数字 chip（相同/过程/衍生） | 保留 |
| primary 行新增 | 无 | 「关联 N ▾」chip，紧贴文件名右边（类似现有「N 副本」chip 的位置）。N = 该 family 中除 primary 外的成员数 |
| 右上角按钮 | 「相似度分析」（直接点直接跑） | 改名「重建相似关系」，点击弹**二次确认对话框** |
| 按钮下方"分析中..."小字 | 有，存在感弱 | 移除，状态统一在二次确认对话框 / 任务面板里展示 |

### 3. 重建相似关系二次确认对话框

```
重建相似关系
─────────────────────────────
约 N 个文件需要重新计算特征
预计耗时 ~M 分钟

上次重建：2026-05-26 11:30，耗时 4 分 12 秒
（首次重建时此行不显示，N = 全部文件）

[ 取消 ]   [ 开始重建 ]
```

数据来源（新增 `POST /analyze/preview` 接口）：
- `cache_miss_count`：stat 后发现 mtime/size 与 DB 不一致的文件数（Spec B 实现后秒级可得）
- `last_run_at`、`last_run_duration`：从 `similarity_task` 表查最近一次成功记录
- `M` = `cache_miss_count` × 上次单文件平均耗时（没有则用 2 秒/文件兜底）

边界：当 `cache_miss_count = 0`（用户没改过文件），显示"无需重算，家族关系已是最新"，"开始重建"按钮置灰。

### 4. 认领弹窗触发逻辑

| 用户操作 | 弹窗形态 |
|---|---|
| 勾 1 行 + 点右上角"认领" | 单文件弹窗 |
| 勾 ≥2 行 + 点右上角"认领" | 批量弹窗（方案 B） |
| 所有勾选行都无家族 | 不弹任何弹窗，直接走原有 `BatchClaim` |
| 勾选行中至少 1 行有家族 | 弹批量弹窗；无家族的行显示「无关联」占位 |
| 用户在系统设置取消了"总是弹窗确认"（等价 `claim_family_skip_dialog = true`） | 跳过所有弹窗，按 `claim_family_default_policy` 直接级联 |

### 5. 单文件认领弹窗

```
┌─ 一并认领相似文件？─────────────────────────────┐
│ 你正在认领：招标方案_v3.docx                       │
│ 分类：个人工作                                       │
│                                                       │
│ 🔗 这个文件还关联到 5 个相似文件                     │
│                                                       │
│ ✓ 相同内容（2 个 · 默认勾选）                       │
│   ☑ 招标方案_v3_备份.docx  /backup/  100%          │
│   ☑ 招标方案-同事版.docx   /downloads/ 100%        │
│                                                       │
│ ☐ 过程版本（2 个 · 默认不勾，按需勾选）             │
│   ☐ 招标方案_v1.docx   /projects/  87%             │
│   ☐ 招标方案_v2.docx   /projects/  92%             │
│                                                       │
│ ☐ 衍生文件（1 个 · 默认不勾）                       │
│   ☐ 招标会议纪要.docx  /meetings/  74%             │
│                                                       │
│ ☐ 以后总是一并认领相似文件，不再询问                 │
│                                                       │
│        [ 仅领此文件 ]  [ 认领 N 个 ]               │

```

要点：
- 默认勾选策略按相似度分级：same_content 勾、process_version 不勾、derived 不勾
- 「下次不再问」勾上后：写入 `system_config.claim_family_skip_dialog = true`，下次跳过弹窗按当前默认 policy 走

### 6. 批量认领弹窗（方案 B）

```
┌─ 认领 30 个文件 · 分类：个人工作 ─────────────────────────┐
│                                                                │
│ 🔗 共识别到 87 个相似文件。批量默认：                          │
│ [ 仅带相同内容（推荐） ▾ ]                                    │
│                                                                │
│ 选中文件（每行可单独调整）                                    │
│                                                                │
│ 招标方案_v3.docx       [关联 5 ▾]  [仅同内容(2) ▾]          │
│   /projects/2024Q4/                                            │
│                                                                │
│ 2024年报-FY24Q4.pptx  [关联 42 ▾]  [仅同内容(8) ▾]          │
│   /reports/                                                    │
│                                                                │
│ 个人简历.docx          [无关联]      [— ]                     │
│   /home/                                                       │
│                                                                │
│ …剩余 27 个                                                    │
│                                                                │
│ ☐ 以后总是按当前默认行为认领，不再询问                         │
│                                                                │
│ 汇总：30 主 + 25 相同（4 个相似已被认领，将跳过）             │
│                                                                │
│        [ 取消 ]              [ 确认认领 55 个 ]               │
└────────────────────────────────────────────────────────────────┘
```

控件行为：

| 控件 | 行为 |
|---|---|
| 顶部"全局默认"下拉 `[仅带相同内容 / 全部带 / 都不带]` | 改变时同步刷新所有行级下拉；**用户手动改过的单行保留**（带"已自定义"标记） |
| 行级 `[关联 N ▾]` chip | N ≤ 20：行内向下折叠展开成员清单；N > 20：右侧侧栏抽屉打开（避免主弹窗布局抖动） |
| 行级下拉 `[仅同内容 N1 / 全选 N / 不带]` | 默认跟随全局；改变后标记该行为"已自定义" |
| 已认领成员 | 列表里灰掉 + 标注"已认领（认领人：张三 · 个人工作）"，不参与本次认领 |
| 底部汇总 | 实时显示当前选择的总数 + 跳过数；分组为 0 的不显示 |
| 「下次不再问」复选框 | 同单文件弹窗，勾上写 `system_config.claim_family_skip_dialog = true` |

### 7. 业务规则

| 规则 | 行为 |
|---|---|
| 分类传播 | 同时被认领的家族成员**继承主文件分类**（保护已有人工分类的成员，见下条） |
| 已认领保护 | 已被认领的成员**不被覆盖**（无论分类是否相同） |
| 跨 tab 处理 | 家族成员跨 tab（工作/新增/历史）也**一并认领**，弹窗里标注"X 个成员属于其他业务来源" |
| 失败处理 | 沿用现有 `BatchClaim` 契约：返回 success + 失败 IDs；前端 toast 显示 |

### 8. 后端 API

| API | 变更 |
|---|---|
| `POST /resources/claim`（BatchClaim） | **不动** |
| `GET /family/:id/members` | **不动** |
| `POST /analyze`（手动重建） | **不动**（给二次确认的"开始重建"用） |
| `POST /analyze/preview` | **新增**：返回 `{cache_miss_count: int, last_run_at: time, last_run_duration_sec: int}` |
| `POST /family/batch-members` | **新增**：接收 `content_signs: [string]`，返回每个 sign 对应的 family + members，避免批量场景 N+1 |

`POST /analyze/preview` 实现：
- 查 `data_distributing` 所有 disable=0 行
- 对每行 `os.Stat()` 拿当前 mtime+size，比 DB 中 `feature_mtime+feature_size`
- 计数不一致 + features 为 NULL 的行
- 从 `similarity_task` 表查最近 succeeded 记录
- 返回结果

### 9. 新增 system_config 项

| key | 值 | 默认 | 说明 |
|---|---|---|---|
| `claim_family_default_policy` | `same_content_only` / `all` / `none` | `same_content_only` | 批量弹窗顶部"全局默认"的初始值；也用于"跳过弹窗"时的级联策略 |
| `claim_family_skip_dialog` | `true` / `false` | `false` | 是否跳过所有弹窗直接按 policy 走 |

### 10. 系统设置页新增卡片

```
┌─ 相似认领默认行为 ─────────────────────────────────┐
│                                                       │
│ 默认对相似家族的处理：                                │
│   ○ 仅认领相同内容（推荐）                           │
│   ○ 认领整个家族（相同 + 过程 + 衍生）              │
│   ○ 不带家族（只认领选中文件）                       │
│                                                       │
│ ☐ 总是弹窗确认（即便已设默认）                       │
│   ↑ 取消勾选此项相当于"下次不再问"                   │
│                                                       │
└───────────────────────────────────────────────────────┘
```

## 数据流（关键路径）

### 批量认领

```
1. 用户勾选 N 行 + 点"认领"
2. 前端调 POST /family/batch-members，传入 N 个 content_sign
3. 前端组装弹窗数据：N 个主文件 + 各自家族成员 + 已认领标记
4. 用户在弹窗里调整后点"确认认领"
5. 前端把"主选 + 勾上的家族成员"合并成 IDs 列表
6. 调用现有 POST /resources/claim（BatchClaim）
7. 后端事务更新所有 IDs 的 claim_status，返回结果
```

### 自动触发 BuildFamilies

```
1. 扫描完成（atomic.go 写完最后一批 data_distributing 后）
2. 检查 runMu 是否被持有
3. 若已被持有（已有任务在跑）→ 跳过
4. 否则 go runAnalysis(taskID)
5. 任务面板/通知中心显示"家族分析中"
```

## 验收标准

- [ ] 扫描完成后无需手动点按钮，进入认领页能看到家族 chip
- [ ] 列表 primary 行有"关联 N ▾" chip，紧贴文件名
- [ ] 勾 1 行点认领，弹单文件弹窗
- [ ] 勾 ≥2 行点认领，弹批量弹窗
- [ ] 批量弹窗里改全局默认下拉，未自定义的行级下拉同步刷新
- [ ] 批量弹窗里改单行下拉，标记"已自定义"，不被后续全局变更覆盖
- [ ] 已认领成员显示但灰掉
- [ ] 跨 tab 的家族成员被一并认领
- [ ] 「重建相似关系」点击弹二次确认，cache_miss_count=0 时"开始重建"灰掉
- [ ] 系统设置可改默认 policy、可切换"总是弹窗"
- [ ] 「下次不再问」勾上后下次跳过弹窗
- [ ] 所有改动通过测试用例（满足 CLAUDE.md「每个任务都要测试用例验证」）

## 风险与回滚

| 风险 | 缓解 |
|---|---|
| 扫描完自动触发 BuildFamilies 在大文件量场景仍慢 | Spec B 直接解决；本 spec 验收前应等 Spec B 至少初步可用 |
| 批量弹窗信息密度高，用户找不到行级控件 | 已用"两步式 + family 折叠"简化；上线后观察用户反馈 |
| 跨 tab 一并认领可能让用户意外 | 弹窗里明确标注"X 个成员属于其他业务来源" |

回滚策略：
- 整个 Spec 改动可通过 feature flag `claim_family_dialog_enabled` 关闭
- 关闭时退回当前直接 `BatchClaim` 流程（不影响数据正确性）
- 列表 chip / 配置项保留（无害）
